# 03. 日々の更新フロー（マキ側）

> LPを編集して本番サイトに反映するまでの、毎回繰り返す手順です。
> 慣れれば10分以内で完了します。

---

## 全体の流れ

```
1. 最新を取り込む（pull）
   ↓
2. 編集用の "枝（ブランチ）" を作る
   ↓
3. VS Codeでファイルを編集
   ↓
4. GitHub Desktop で「コミット」する（=変更を確定する操作）
   ↓
5. GitHub に "push"（=変更を送る）
   ↓
6. ブラウザで PR（プルリクエスト）を作る
   ↓
7. Vercel が自動でプレビューURLを作る → そこで見た目を確認
   ↓
8. ハリーがレビュー・OKすれば main にマージ → 本番反映
```

**なぜこんなに段階があるの？**
- いきなり本番を触ると、間違いが起きた時に戻せません
- "枝（ブランチ）" で作業 → "本流（main）" に合流させる、という流れにすることで、ハリーが事前確認できます

---

## Step 1: 最新を取り込む（pull）

毎回最初にやってください。ハリーが作業した変更を取り込みます。

### 1-1. GitHub Desktop で main ブランチに切り替え

GitHub Desktop の画面上部、**Current branch** が `main` になっているか確認。
違うブランチなら、クリックして `main` を選択。

`[📸スクショ位置: Current branch 表示]`

### 1-2. Fetch + Pull

1. 画面右上の **`Fetch origin`** ボタンをクリック
2. ボタンが **`Pull origin`** に変わったら、もう一度クリック
3. 更新が完了します

`[📸スクショ位置: Fetch/Pull ボタン]`

---

## Step 2: 編集用の枝（ブランチ）を作る

### 2-1. 新しいブランチを作る

GitHub Desktop の画面上部、**Current branch** をクリック → **`New branch`** ボタン

`[📸スクショ位置: New branch ボタン]`

### 2-2. ブランチ名を入力

分かりやすい名前を付けます。形式は **`fix/何を直したか`** または **`feat/何を追加したか`**。

例:
- `fix/header-text` （ヘッダーのテキスト修正）
- `fix/menu-price` （メニュー価格の修正）
- `feat/add-instagram-feed` （Instagramフィードを追加）
- `update/manga-section1` （マンガセクション1の画像差し替え）

入力したら **`Create branch`** をクリック。

`[📸スクショ位置: New branch ダイアログ]`

---

## Step 3: ファイルを編集する

### 3-1. VS Code でフォルダを開く

GitHub Desktop の **Repository** メニュー → **Open in Visual Studio Code**

### 3-2. 編集できる主な場所

| やりたいこと | 編集するファイル/フォルダ |
|------------|----------------------|
| TOPページのテキスト変更 | `src/pages/Home.tsx` |
| アカデミーページのテキスト変更 | `src/pages/Academy.tsx` |
| ヘッダーのテキスト | `src/components/layout/Header.tsx` |
| フッターのテキスト | `src/components/layout/Footer.tsx` |
| 画像差し替え | `public/images/` （同じファイル名で上書き or 新規追加） |
| マンガ画像差し替え | `public/manga/salon/section*/` |
| プライバシーポリシー | `src/pages/Privacy.tsx` |
| 特商法表記 | `src/pages/Law.tsx` |

### 3-3. テキスト編集の基本

VS Code の左サイドバーでファイルを選んでクリック → 中央の編集エリアに内容が表示 → 編集 → `Ctrl + S` で保存。

`[📸スクショ位置: VS Code 編集画面]`

**注意点**:
- `<div>` `<p>` などの `<...>` で囲まれた部分は構造を表すコードです。**触らない**でください。
- テキストを書き換えるときは、`>` と `<` の間の文字だけ変更します。

例:
```html
<h1>古いタイトル</h1>     ← <h1>と</h1>はそのまま
       ↓
<h1>新しいタイトル</h1>   ← 真ん中だけ変える
```

### 3-4. 画像の差し替え

1. エクスプローラーで `public/images/` フォルダを開く
2. 古い画像と**同じファイル名**で新しい画像を上書き保存
3. または新しいファイル名で追加 → コード側の参照パスを変更（要相談）

---

## Step 4: GitHub Desktop で「コミット」する

「コミット」= 変更内容を「ここまでの作業を区切る」として確定する操作。

### 4-1. 変更ファイルを確認

GitHub Desktop に戻ると、画面左に「変更されたファイル」が一覧表示されています。
チェックマークが付いているファイルがコミット対象です。

`[📸スクショ位置: 変更ファイル一覧]`

中央には変更の差分（赤=削除、緑=追加）が表示されます。

### 4-2. コミットメッセージを書く

画面左下の入力欄に、何を変えたかを書きます。

| 入力欄 | 書く内容 |
|--------|---------|
| Summary（必須・1行） | 例: `ヘッダーのキャッチコピーを更新` |
| Description（任意） | 詳細が必要なら箇条書きで |

### 4-3. Commit ボタンを押す

**`Commit to <ブランチ名>`** という青いボタンをクリック。

`[📸スクショ位置: Commit ボタン]`

---

## Step 5: GitHub に push（=変更を送る）

### 5-1. Push origin ボタン

コミット直後、画面上部に **`Push origin`** という青いボタンが現れます。クリック。

`[📸スクショ位置: Push origin ボタン]`

数秒で完了します。

---

## Step 6: PR（プルリクエスト）を作る

ブラウザに移動します。

### 6-1. Create Pull Request ボタンが出る

GitHub Desktop の画面上部に **`Create Pull Request`** ボタンが表示されるので、クリック。

`[📸スクショ位置: Create Pull Request ボタン]`

ブラウザで GitHub のページが自動で開きます。

### 6-2. PR の内容を書く

| 項目 | 書き方 |
|------|--------|
| Title | コミットメッセージと同じか、もう少し丁寧に（例: `ヘッダーのキャッチコピー更新 → "肌が変われば...." に` ） |
| Description | 変更内容の説明・スクショ・気になる点・確認してほしい点 |
| Reviewers | 右サイドバーの **Reviewers** → `harry-n2` を指定 |

`[📸スクショ位置: PR作成画面]`

### 6-3. Create pull request ボタン

緑色の **`Create pull request`** をクリック。

これでハリーに通知が届きます。

---

## Step 7: Vercel プレビューで確認

PR を作ると、数分以内に Vercel が自動でプレビューURLを生成します。

### 7-1. PR ページにコメントが追加される

PR画面の下のほうに、**vercel[bot]** からのコメントが追加されます:

```
Visit the preview URL for this PR (updated for commit xxxxxx):
https://makisalon-git-fix-header-text-xxxxx.vercel.app
```

`[📸スクショ位置: Vercelプレビューコメント]`

### 7-2. プレビューを開いて確認

そのURLをクリック → ブラウザで「自分の変更が反映されたサイト」が見えます。
意図通りになっているか確認してください。

### 7-3. 問題があれば修正

プレビューで「思っていたのと違う」場合:
1. VS Code でさらに編集
2. GitHub Desktop で再度 Commit → Push
3. **同じPRに自動で追加されます**（新しいPRを作る必要なし）
4. プレビューURLが自動更新される

---

## Step 8: ハリーレビュー → 本番反映

### 8-1. ハリーがレビュー

ハリーが PR を見て、コメントしたり、OK だったら承認したりします。
コメントが付いたら、その内容に従って修正してください。

### 8-2. main にマージ

ハリーが承認すると、PR画面の **`Merge pull request`** ボタンが緑色になります。
このマージは **ハリーが押す** ことが多いですが、マキさんが押しても問題ありません（マキさんも編集権限があるので）。

### 8-3. 本番反映

main にマージされると、Vercel が自動で本番デプロイを実行します。
数分以内に https://makisalon.vercel.app/ に反映されます。

`[📸スクショ位置: 本番反映の確認]`

---

## ブランチの後片付け

PR がマージされたら、使い終わったブランチは削除して構いません。

### GitHub上で削除

PR画面の **`Delete branch`** ボタンをクリック → リモートから削除されます。

### ローカル（GitHub Desktop）で削除

GitHub Desktop の **Current branch** → 該当ブランチを右クリック → **`Delete...`**

ローカルもmainに戻して、次回作業時はまた Step 1 から始めます。

## よくある質問

### Q. 編集中に間違えた・元に戻したい

- ファイルを保存（Ctrl+S）していなければ、`Ctrl + Z` で1つ前に戻る
- 保存してしまったら、GitHub Desktop の変更ファイル一覧で該当ファイルを右クリック → **`Discard changes`** で元に戻る

### Q. ブランチ名や PRタイトルは日本語でいいの？

ブランチ名は **半角英数字+ハイフン** で書きます（例: `fix/header-text`）。
PRタイトルや説明は **日本語OK** です。

### Q. 同時にハリーが作業していたら大丈夫？

GitHub が自動で合流（マージ）してくれます。
ただし、同じファイルの同じ行を両方が直していると「コンフリクト（衝突）」が発生します。
その場合は `04_緊急時マニュアル.md` のコンフリクト対処を見るか、ハリーに連絡してください。

### Q. pushしたけど Vercel プレビューが出てこない

5分待ってもコメントが追加されない場合:
1. PR画面をブラウザで再読み込み（F5）
2. それでも出ない場合はハリーに連絡（Vercel側の設定確認が必要かも）

### Q. コミットメッセージは何を書けばいい？

「何を変えたか」が一目で分かれば十分です。完璧でなくてOK。

| 良い例 | 改善例 |
|--------|--------|
| `update` | `ヘッダー画像を新しいものに差し替え` |
| `修正` | `アカデミーページの料金表記を5,500円に修正` |
| `aaa` | `お問い合わせ先のメールアドレス変更` |
