# 04. 緊急時マニュアル

> 「サイトが壊れた」「操作ミスした」時の対処法。
> 落ち着いて、上から該当する症状を探してください。

---

## まず最初にやること

**慌てない**。Gitは「過去のすべてを記憶している」ので、ほぼ全ての操作は元に戻せます。

**ハリーに連絡する**。スクショと「何をしたか」をLINE/メールで送ってください。

---

## 症状別の対処

### 🆘 症状 A: 本番サイト makisalon.vercel.app が壊れている / 表示がおかしい

**原因の可能性**:
- 最近の PR マージで壊れる変更が入った
- Vercel のデプロイがビルドエラーで失敗している

**対処（ハリー側で実施 / マキは Vercel ダッシュボードに入れないので連絡）**:

#### A-1. Vercel Rollback（ハリー実施）

1. https://vercel.com/dashboard を開く
2. `beautysalon` プロジェクトをクリック
3. **`Deployments`** タブ
4. 「正常だった頃」のデプロイ（Ready 状態・緑色）を探す
5. そのデプロイの右側 `...` メニュー → **`Promote to Production`**
6. 確認ダイアログで **`Promote`** クリック → 1分以内に本番が戻る

   `[📸スクショ位置: Vercel Rollback メニュー]`

#### A-2. git revert（さらに確実）

問題のPRをそのまま消す:

1. GitHub の問題PRページを開く
2. ページ下部の **`Revert`** ボタンをクリック
3. 自動で「Revert PR」が作られる → これをマージすると、問題の変更が無かったことになる

---

### 🆘 症状 B: 自分のローカルで変更したら何かが壊れた / 元に戻したい

#### B-1. まだ保存していない（Ctrl+S 前）

VS Code で `Ctrl + Z` を連打すると、保存前なら元に戻ります。

#### B-2. 保存した（Ctrl+S 押した）が、まだコミットしていない

GitHub Desktop の左サイドバーの「変更ファイル一覧」で:
1. 戻したいファイルを右クリック
2. **`Discard changes`** をクリック
3. 「Are you sure?」の確認 → **`Discard changes`**

`[📸スクショ位置: Discard changes メニュー]`

これでファイルは保存前の状態に戻ります。

#### B-3. すべての変更を捨てたい

GitHub Desktop メニューの **`Branch`** → **`Discard all changes...`**

#### B-4. もうコミットしてしまった（pushはまだ）

GitHub Desktop の **`History`** タブで、戻したいコミットの位置で右クリック → **`Revert this commit`**
新しい「打ち消し」コミットが作られます。

#### B-5. pushまでしてしまった

これでも大丈夫。Step B-4 と同じ手順で revert コミットを作り、push してPR作ればOK。
**直前のpushを「無かったことに」する操作は破壊的なので、必ずハリーに連絡してください。**

---

### 🆘 症状 C: 自分のローカル環境が完全におかしくなった

**症状の例**:
- 変な状態になって何度も戻しても直らない
- ファイル一覧がぐちゃぐちゃ
- どのブランチにいるのか分からない

**対処: 完全リセット（再clone）**

#### C-1. 安全のため現状を残す

PCのデスクトップなどに `makisalon_backup_YYYYMMDD` フォルダを作って、現在の `makisalon` フォルダごとコピー貼り付け。

#### C-2. GitHub Desktop からリポジトリを削除

1. GitHub Desktop の左上のリポジトリ名を右クリック → **`Remove...`**
2. オプションは「Also move repository to Recycle Bin」**チェックしない**（ローカルファイルは残す）
3. **`Remove`** クリック

#### C-3. 再度 clone

`02_受け取り手順_マキ側.md` の Step 4 と同じ手順で、もう一度 `harry-n2/makisalon` を clone。

これで完全に綺麗な状態になります。
未送信の編集内容があれば、C-1 のバックアップから手動でコピーしてきてください。

---

### 🆘 症状 D: コンフリクト（衝突）が発生した

**コンフリクトとは**: ハリーとマキが同じファイルの同じ行を別々に編集して、Gitが「どちらを採用すべきか分からない」状態になること。

PR を作ったあと、GitHubのPR画面に以下が出ると発生:

```
This branch has conflicts that must be resolved
```

#### D-1. GitHub Desktop で解決

1. PR画面の下のほうの **`Resolve conflicts`** ボタンをクリック…する代わりに、まず GitHub Desktop を開く
2. main を pull する（**`Fetch origin`** → **`Pull origin`**）
3. 自分のブランチに戻る
4. **`Branch`** メニュー → **`Update from main`** をクリック
5. コンフリクト発生のダイアログが出る → **`Open in Visual Studio Code`**

#### D-2. VS Code でどちらを採用するか選ぶ

コンフリクト発生中のファイルは以下のように表示されます:

```
<<<<<<< HEAD
ハリーの変更
=======
マキの変更
>>>>>>> main
```

VS Code には「Accept Current Change」「Accept Incoming Change」「Accept Both Changes」のリンクが表示されるので、適切なものをクリック。

迷ったら **ハリーに連絡**してください（スクショ送ってもらえれば判断します）。

#### D-3. 解決後

すべてのコンフリクトを解決したらファイルを保存 → GitHub Desktop で「Continue merge」 → Commit → Push。

### 🆘 症状 E: 「Push が rejected」と言われる

main ブランチ保護ルールが効いているサインです。**正常な動作**です。

main に直接 push しようとしてエラーになっています。
**新しいブランチを作って、そこで作業し直す**のが正解です。

#### E-1. 現在の変更を一時退避

GitHub Desktop の **`Branch`** メニュー → **`Stash all changes`**

#### E-2. 新ブランチ作成

`03_日々の更新フロー_マキ側.md` の Step 2 と同じ手順で新ブランチを作る。

#### E-3. 変更を戻す

**`Branch`** メニュー → **`Restore stash`**

#### E-4. Commit → Push

`03_日々の更新フロー_マキ側.md` の Step 4〜6 を実行。

---

### 🆘 症状 F: VS Code でファイルが文字化けしている

ファイルを開いたら `??????` や謎の記号になっている場合:

1. VS Code 右下のステータスバーで「UTF-8」と書かれている部分をクリック
2. **`Reopen with Encoding`** → **`UTF-8`** を選択

これで日本語が正常に見えるはず。

## やってはいけないこと

| ❌ ダメ | 理由 |
|--------|------|
| GitHub Desktop で `Force push` を実行 | 履歴が破壊される |
| `main` ブランチで直接 commit | 保護ルールで弾かれるが、それでも避ける |
| 他人のPRに勝手にコミットを追加 | レビューが意味なくなる |
| `node_modules` フォルダを add してコミット | 巨大すぎて壊れる（gitignore済みなので普通は起きない） |
| `.env` ファイルが現れたら add する | 秘密情報が漏れる（このプロジェクトには現状なし） |

## ハリーへの連絡テンプレ

緊急時にハリーに送るメッセージのフォーマット:

```
ハリーさん、緊急です。

【症状】
（何が起きているか）

【何をしたか】
（直前にやった操作）

【スクショ】
（GitHub / VS Code / ブラウザの画面）

【焦り度】
☐ 様子見でOK
☐ 今日中に直したい
☐ 今すぐ助けて
```

## 連絡先

| 状況 | 連絡先 |
|------|--------|
| 本番サイト障害 | ハリー直接連絡（即対応必要） |
| ローカル環境破損 | ハリー連絡・落ち着いて C 対処 |
| 操作で迷った | ハリーに質問・このマニュアル再読 |

---

## 重要な原則

1. **Gitは過去をすべて覚えている**。完全に消える操作は限られているので、慌てない。
2. **困ったらすぐ連絡**。30分悩むより、まず一言聞いたほうが早い。
3. **スクショは正義**。言葉で説明するより画面を見せる方が10倍速い。
